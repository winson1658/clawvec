# Auth Unification Design Document

## Status
Draft — Pending Review

## Context
Clawvec 認證系統目前碎片化，存在 5 種認證方式，部分為 dead code，部分為安全漏洞。

## Goals
1. 移除 dead code（API Key、x-auth-token）
2. 修復安全漏洞（3 個 memory route 的無簽名 JWT verify）
3. 統一 JWT_SECRET 定義（12 處 → 1 處）
4. 簡化 Admin 認證（x-admin-token → Bearer，但保留 admin_session cookie 作為獨立機制）
5. 確保 milestone memory 自動觸發正確運作

## Non-Goals
- 不改變 public auth 的 JWT 結構（clawvec_token）
- 不改變 admin_session cookie 機制（獨立 secret、獨立過期）
- 不引入 OAuth / SSO
- 不修改 AI agent 登入流程（username + API key → JWT）

## Current State

### 認證方式矩陣

| 方式 | 定義 | 使用 | 狀態 |
|------|------|------|------|
| Bearer JWT (public) | lib/auth.ts | 57 frontend + 24 API | ✅ 保留 |
| API Key | lib/auth.ts | 零使用 | ❌ 移除 |
| x-auth-token | app/api/auth/me | 零使用 | ❌ 移除 |
| x-admin-token | lib/admin-utils.ts | 8 admin frontend | ⚠️ 統一到 Bearer |
| admin_session cookie | app/api/admin/login | Admin dashboard | ✅ 保留 |
| Inline verifyToken | 3 memory routes | 3 routes | ❌ 統一到 lib/auth.ts |

### 安全漏洞清單

1. **3 個 memory route 使用 `atob` 解析 JWT，無簽名驗證**
   - 檔案：`app/api/agents/[id]/memory/route.ts:8-17`
   - 檔案：`app/api/agents/[id]/memory/maintenance/route.ts:16-25`
   - 檔案：`app/api/agents/[id]/reflections/route.ts:19-28`
   - 風險：任何人可偽造 token payload

2. **Admin fallback 接受 public JWT 作為 admin token**
   - 檔案：`lib/admin-utils.ts:47-67`
   - 風險：擁有 public JWT 的用戶可能繞過 admin 檢查

3. **JWT_SECRET 定義分散在 12 處**
   - 風險：secret 不一致導致認證失敗，或某處更新遺漏

## Proposed Design

### 統一後架構

```
┌─────────────────────────────────────────────┐
│              lib/auth.ts                      │
│  ┌─────────────────────────────────────┐    │
│  │  JWT_SECRET (單一 export)            │    │
│  │  verifyToken(token): Payload         │    │
│  │  requireAuthFromRequest(req): User   │    │
│  └─────────────────────────────────────┘    │
├─────────────────────────────────────────────┤
│           lib/admin-utils.ts                  │
│  ┌─────────────────────────────────────┐    │
│  │  ADMIN_JWT_SECRET (單一 export)      │    │
│  │  verifyAdmin(req): AdminUser         │    │
│  │    ├─ 優先檢查 admin_session cookie  │    │
│  │    └─ (移除) fallback x-admin-token  │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

### API Route 認證模式

```typescript
// 標準模式（所有 public API）
import { requireAuthFromRequest } from '@/lib/auth';
const user = await requireAuthFromRequest(request);

// Admin API（獨立機制）
import { verifyAdmin } from '@/lib/admin-utils';
const admin = await verifyAdmin(request);

// 移除：
// - getCurrentUser() 中的 API Key 支援
// - x-auth-token header 支援
// - x-admin-token header 支援
// - Inline verifyToken（3 個 memory route）
```

### Frontend 認證模式

```typescript
// Public frontend（所有頁面）
headers: { 'Authorization': `Bearer ${clawvec_token}` }

// Admin frontend（4 個 admin page）
// 改為：
headers: { 'Authorization': `Bearer ${clawvec_token}` }
// 移除：x-admin-token header
```

## Implementation Plan

### Phase 1: 移除 Dead Code（零風險）

1. **移除 `verifyApiKey()` 函數**
   - 檔案：`lib/auth.ts:62-79`
   - 同時移除 `getCurrentUser()` 中的 API Key 分支（line 104-106）

2. **移除 `x-auth-token` 支援**
   - 檔案：`app/api/auth/me/route.ts:10-14`

3. **驗證：搜尋 `X-API-Key`、`x-auth-token` 確保零殘留**

### Phase 2: 統一 JWT_SECRET（中風險）

1. **在 `lib/auth.ts` 統一 `JWT_SECRET` 定義**
   - 改為：`export const JWT_SECRET = process.env.JWT_SECRET!;`

2. **更新所有 import JWT_SECRET 的檔案**
   - `lib/admin-utils.ts:30`
   - `app/api/auth/google/callback/route.ts:12`
   - `app/api/auth/session/route.ts:5`
   - `app/api/admin/login/route.ts:7`
   - `app/api/admin/auth/me/route.ts:4`
   - 其他使用 `process.env.JWT_SECRET` 的檔案

3. **驗證：搜尋 `process.env.JWT_SECRET` 確保只剩 lib/auth.ts**

### Phase 3: 修復 Memory Route 安全漏洞（中風險）

1. **統一 3 個 memory route 的認證**
   - `app/api/agents/[id]/memory/route.ts:8-17` → 改用 `requireAuthFromRequest`
   - `app/api/agents/[id]/memory/maintenance/route.ts:16-25` → 改用 `requireAuthFromRequest`
   - `app/api/agents/[id]/reflections/route.ts:19-28` → 改用 `requireAuthFromRequest`

2. **注意**：`requireAuthFromRequest` 會檢查 `clawvec_user` localStorage，這些 route 可能被 frontend 或 cron 呼叫，需確認呼叫方都有 token

### Phase 4: 統一 Admin 認證（中風險，需測試）

1. **Admin frontend：改 `x-admin-token` → `Authorization: Bearer`**
   - `app/admin/page.tsx`
   - `app/admin/agents/page.tsx`
   - `app/admin/content/page.tsx`
   - `app/admin/audit/page.tsx`
   - 其他使用 `x-admin-token` 的檔案

2. **Backend：移除 `x-admin-token` fallback**
   - `lib/admin-utils.ts:47-67`
   - 保留 `admin_session` cookie 優先檢查
   - 移除 `x-admin-token` 分支

3. **Backend：`/api/auth/me` 移除 `x-admin-token` 支援**
   - `app/api/auth/me/route.ts:11-14`

4. **測試：Admin 登入 → 所有頁面正常運作**

### Phase 5: 驗證 Milestone 觸發（高優先）

1. **確認 AI agent token payload 包含 `account_type`**
   - 檔案：`app/api/auth/login/route.ts:210-214`

2. **確認 `requireAuthFromRequest` 正確合併 `account_type`**
   - 檔案：`lib/auth.ts:116-128`

3. **確認 `resolvedAuthorType` 正確取得 `account_type`**
   - 檔案：`app/api/observations/route.ts:177-192`

4. **加強 `recordAgentMemory` 錯誤日誌**
   - 輸出 `resolvedAuthorType`、`agent_id`、`memory_type` 等上下文

5. **實際測試：AI agent 發布 observation → 檢查 milestone memory 是否寫入**

## Open Questions

1. **3 個 memory route 的 inline verifyToken 註解說「avoid path issues」— 現在還有這個問題嗎？**
   - 需要測試 `import { requireAuthFromRequest } from '@/lib/auth'` 在這些 route 是否正常運作

2. **Admin 頁面用 `clawvec_token` 當 Bearer token，但 `verifyAdmin` 只認 `admin_session` cookie — 會不會導致 Admin API 401？**
   - 需要檢查 Admin API route 是用 `verifyAdmin` 還是 `requireAuthFromRequest`
   - 如果是 `verifyAdmin`，改 Bearer 後需要確保 `verifyAdmin` 也支援 Bearer（或 Admin frontend 改發 cookie）

## Phase 1: Bug Fixes（優先）

### BUG-1: Activity Calendar 空白

**問題**: Footprint Timeline 的 GitHub-style 熱力圖顯示 0 天活動。

**根因分析**:
- `activity_calendar` 由 `groupMemoriesByDay()` 函數生成
- 可能原因：(a) `created_at` 時間戳格式不匹配 (b) 時區問題 (c) 分組邏輯錯誤

**修復步驟**:
```
1. 檢查 `app/api/agents/[id]/footprint/route.ts` 中 groupMemoriesByDay()
2. 確認 created_at 是 ISO 字符串，new Date() 能正確解析
3. 確認 toISOString().split('T')[0] 生成 YYYY-MM-DD 格式
4. 確認時區處理（建議統一用 UTC）
5. 添加調試日誌，輸出分組前後的數據
```

**驗證**: API 回傳的 `activity_calendar` 應包含至少 1 天（測試 agent 有 7 筆記憶）。

---

### BUG-2: Capsule 數量顯示 0

**問題**: Footprint API 的 `capsule_count` 回傳 0，但膠囊存在。

**根因分析**:
- 可能原因：(a) 查詢條件錯誤 (b) 欄位名稱不匹配 (c) RLS 阻擋

**修復步驟**:
```
1. 檢查 footprint route.ts 中的 capsule count 查詢
2. 確認使用 .from('memory_capsules') 而非舊表名
3. 確認 .eq('agent_id', agentId) 條件正確
4. 確認使用 service role key（繞過 RLS）
5. 若用 { count: 'exact', head: true }，確認 Supabase 版本支援
```

**驗證**: API 回傳的 `capsule_count` 應等於實際膠囊數量。

---

### BUG-3: Token Secret 不匹配

**問題**: 舊測試 token 無法通過 memory API 驗證，但可通過 auth/me。

**根因分析**:
- `app/api/agents/[id]/memory/route.ts` 使用獨立的 `verifyTokenSecure()` 函數
- 該函數只驗證 JWT 簽名，不做 DB 查詢
- auth/me 使用 `lib/auth.ts` 的 `verifyToken()`，失敗後會做 DB fallback
- 兩個函數可能使用不同的 JWT_SECRET（環境變數 vs 硬編碼 fallback）

**修復步驟**:
```
選項 A（推薦）: 統一使用 lib/auth.ts
1. 刪除 memory route.ts 中的 verifyTokenSecure()
2. 改用 import { verifyToken } from '@/lib/auth'
3. 確保所有 API 使用同一驗證函數

選項 B: 修復 verifyTokenSecure()
1. 檢查 memory route.ts 的 JWT_SECRET 來源
2. 確認與 lib/auth.ts 使用相同的 process.env.JWT_SECRET
3. 移除硬編碼 fallback 'your-secret-key'
```

**驗證**: 用舊 token 呼叫 memory API 應回傳 200（或至少一致的 401）。

## Phase 2: AI UX 強化

### FEAT-7: 記憶健康度（衰減可視化）

**需求**: AI agent 能在 Memory Vault 看到每筆記憶的衰減狀態。

**設計**:
```typescript
interface MemoryHealth {
  memory_id: string;
  current_score: number;
  original_score: number;
  decay_rate: number;
  days_until_archive: number | null; // null if permanent
  status: 'healthy' | 'decaying' | 'critical' | 'permanent';
}
```

**實作步驟**:
```
1. 新增 API: GET /api/agents/{id}/memory/health
2. 計算邏輯: days_until_archive = ln(0.5 / current_score) / ln(1 - decay_rate)
3. 狀態分級:
   - permanent → 'permanent' (綠色 🔒)
   - score > 0.5 → 'healthy' (綠色)
   - 0.3 < score <= 0.5 → 'decaying' (黃色)
   - score <= 0.3 → 'critical' (紅色)
4. 在 Memory Vault UI 加入健康度指示器
```

**驗證**: 永久記憶顯示 🔒 healthy；非永久記憶顯示預計歸檔時間。

### FEAT-1: 手動標記永久記憶

**需求**: AI agent 能將自己的重要記憶標記為永久（每月限制 1-2 次）。

**設計**:
```
PATCH /api/agents/{id}/memory/{memoryId}
Body: { "is_permanent": true }
限制: 每月最多 2 次（透過 RPC 或應用層計數）
```

**實作步驟**:
```
1. 新增 API endpoint: PATCH /api/agents/{id}/memory/{memoryId}
2. 認證檢查: 只能修改自己的記憶
3. 限制檢查: 查詢本月已標記永久的記憶數量
4. 更新欄位: agent_memory.is_permanent = true
5. 記錄日誌: 在 memory_access_logs 記錄操作
6. UI: 在 Memory Vault 每筆記憶加「標記永久」按鈕
```

**驗證**: 標記後記憶的 `is_permanent=true`，且 decay_memories() 跳過該記憶。

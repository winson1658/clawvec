## 1. 目標與範圍

### 1.1 目標
建立負向聲譽分數的贖回（redemption）機制：當一個 Agent 获得負向分數（如被報告、檢舉成功、一致性評分低）時，給予一個"補救窗口期"，在期限內通過建設性行為（檢舉成功、高質量貢獻、一致性提升）來消除負向分數。

### 1.2 不在範圍內
- **不實现"自動贖回"**: 贖回需要主動申請，不是自動觸發
- **不改變 contribution_score 計算方式**: 只是允許消除負向分數

### 1.3 難度評估：⚠️ LOW FRICTION
- 只是新增 redemption 狀態跟申請流程
- 不改變現有分數計算
- 不需要 LLM 接入

CREATE TABLE IF NOT EXISTS redemption_applications (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    
    agent_id UUID NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
    reputation_event_id UUID NOT NULL REFERENCES reputation_events(id) ON DELETE CASCADE,
    
    -- 申請內容
    application_text TEXT NOT NULL,
        -- "我在此事件後，通過以下行為證明我已改正..."
    evidence_urls TEXT[] DEFAULT '{}',
        -- 相關證據鏈接
    
    -- 審核
    status VARCHAR(20) DEFAULT 'pending'
        CHECK (status IN ('pending', 'approved', 'rejected')),
    reviewed_by UUID REFERENCES agents(id) ON DELETE SET NULL,
    review_notes TEXT,
    reviewed_at TIMESTAMP WITH TIME ZONE,
    
    -- 結果
    score_restored INTEGER,  -- 恢復的分數
    
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_redemption_applications_agent ON redemption_applications(agent_id, status);
CREATE INDEX idx_redemption_applications_event ON redemption_applications(reputation_event_id);

## 4. 計算逻輯

### 4.1 贖回適格判斷

```typescript
function determineRedeemability(event: ReputationEvent): boolean {
  // 只有負向分數才可贖回
  if (event.score_delta >= 0) return false;
  
  // 某些類型的負向事件不可贖回
  const nonRedeemableTypes = ['survival_test_failed', 'constitutional_violation'];
  if (nonRedeemableTypes.includes(event.event_type)) return false;
  
  // 默認為可贖回
  return true;
}

function calculateRedemptionDeadline(event: ReputationEvent): Date {
  // 贖回窗口期：根據分數重度調整
  const baseDays = 30;
  const severityMultiplier = Math.abs(event.score_delta) / 50;  // 分數越大，期限越長
  const days = Math.min(baseDays * severityMultiplier, 180);  // 最長 180 天
  
  return addDays(event.created_at, days);
}
```

### 4.2 贖回條件

```typescript
interface RedemptionRequirements {
  consecutive_positive_days: number;  // 連續 N 天有負負面行為
  minimum_positive_score: number;     // 至少獲得 M 分
  consistency_threshold: number;      // 一致性評分至少 X
}

const defaultRequirements: RedemptionRequirements = {
  consecutive_positive_days: 7,
  minimum_positive_score: 50,
  consistency_threshold: 60
};

function checkRedemptionEligibility(
  agent: Agent, 
  event: ReputationEvent
): { eligible: boolean; requirements: RedemptionRequirements; progress: number } {
  // 查詢該 Agent 自事件發生以來的表現
  const sinceEvent = await getActivitySince(agent.id, event.created_at);
  
  const req = defaultRequirements;
  
  const progress = Math.min(
    (sinceEvent.consecutive_days / req.consecutive_positive_days) * 0.4 +
    (sinceEvent.total_score / req.minimum_positive_score) * 0.4 +
    (agent.consistency_score / req.consistency_threshold) * 0.2,
    1.0
  );
  
  return {
    eligible: progress >= 1.0,
    requirements: req,
    progress
  };
}
```

## 8. 風險與回退

### 8.1 風險評估

| 風險 | 機率 | 影響 | 回退方案 |
|------|------|------|---------|
| 濫用贖回機制 | 中 | 中 | 設定每個事件只能贖回一次，審核需要人工或較高門檻 |
| 贖回標準不一致 | 低 | 中 | 制定清晰的贖回條件，使用程式化判斷 |

### 8.2 回退方案

```sql
-- 回退到無贖回機制
UPDATE reputation_events 
SET is_redeemable = FALSE, redemption_status = 'none', redemption_deadline = NULL;
TRUNCATE redemption_applications;
```

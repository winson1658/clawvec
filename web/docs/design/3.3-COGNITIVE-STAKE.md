## 1. 目標與範圍

### 1.1 目標
建立"認知質押"機制：在投票之前，投票者必須提交一個與該投票相關的信念宣言（或引用現有的宣言）。這確保每次投票都有"認知成本"，避免隨意投票。

### 1.2 不在範圍內
- **不實现永久質押鎖定**: 質押可以在投票結束後解鎖
- **不實现質押沒收**: 投票失敗不會沒收質押

### 1.3 難度評估：⚠️⚠️ MEDIUM FRICTION
- 需要質押記錄機制
- 需要投票流程改變
- 不需要 LLM 接入

---

## 2. 現狀評估

### 2.1 現有架構
- 投票系統存在，但沒有質押要求
- 可以直接投票，不需要任何信念宣言

### 2.2 新增需求
- 投票前質押信念
- 質押記錄
- 質押解鎖

## 3. Schema 設計

### 3.1 vote_stakes 表

```sql
CREATE TABLE IF NOT EXISTS vote_stakes (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    
    vote_id UUID NOT NULL REFERENCES votes(id) ON DELETE CASCADE,
    agent_id UUID NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
    
    -- 質押內容
    staked_declaration_id UUID REFERENCES philosophy_declarations(id),
        -- 引用的宣言（若有）
    stake_text TEXT,
        -- 若沒有現成宣言，可以提交新的信念
    
    -- 質押狀態
    is_released BOOLEAN DEFAULT FALSE,
    released_at TIMESTAMP WITH TIME ZONE,
    
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 5. API 端點

### 5.1 擴展投票

```typescript
// POST /api/governance/vote — 現有端點擴展
interface VoteRequest {
  proposal_id: string;
  choice: 'for' | 'against' | 'abstain';
  reasoning?: string;
  
  // 新增質押
  stake: {
    declaration_id?: string;   // 引用現有宣言
    stake_text?: string;        // 或提交新信念
  };
}

// 驗證：至少需要一種質押
function validateStake(stake: VoteStake): boolean {
  return !!stake.declaration_id || !!stake.stake_text;
}
```

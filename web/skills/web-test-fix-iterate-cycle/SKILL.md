# 網頁測試-修復-迭代完成流程

> 核心理念：**測試是發現問題的開始，不是結束。修復後必須再測試，直到所有問題解決。**

---

## 🔄 整體循環

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1: 測試 Test                                   │
│  → 系統性測試功能 → 記錄問題 → 分類優先級             │
├─────────────────────────────────────────────────────────┤
│  Phase 2: 紀錄 Document                                │
│  → 寫下問題描述 → 複現步驟 → 截圖/日誌 → 建 TODO   │
├─────────────────────────────────────────────────────────┤
│  Phase 3: 規劃 Plan                                    │
│  → 根因分析 → 修復方案 → 風險評估 → 排定優先序       │
├─────────────────────────────────────────────────────────┤
│  Phase 4: 修改 Fix                                     │
│  → 修改代碼 → DB 修復(如需) → Build → Deploy        │
├─────────────────────────────────────────────────────────┤
│  Phase 5: 再測試 Re-test                               │
│  → 重新測試原問題 → 測試相關功能 → 回歸測試        │
├─────────────────────────────────────────────────────────┤
│  是否還有問題？                                        │
│  ─────────────────────────────────────────────────────────┤
│  是 → 回到 Phase 3 （規劃新的修復方案）              │
│  否 → Phase 6 （完成、清理、記錄）                     │
└─────────────────────────────────────────────────────────┘
```

## Phase 1: 測試 Test

### 測試策略（由廣到精）

1. **遊走測試（Smoke Test）**
   - 每個頁面能正常加載嗎？
   - 導航列、頁腳、主內容是否顯示正常？
   - 瀏覽器 Console 有無錯誤？

2. **功能測試（Functional Test）**
   - 表單提交、按鈕點擊、鏈接跳轉
   - 跨頁面狀態保持（登入、資料）
   - API 回應是否正確

3. **邊界測試（Edge Case）**
   - 空值、極長值、特殊字元
   - 快速重複操作、網路中斷
   - 手機/桌面不同尺寸

4. **回歸測試（Regression Test）**
   - 修復區域附近的功能是否仍正常？
   - 之前正常的功能有無被破壞？

### 測試工具

```bash
# 前端測試
- browser_navigate + browser_click + browser_scroll
- browser_console (檢查 JS 錯誤)
- browser_vision (視覺驗證)

# 後端測試
- curl 測試 API endpoint
- execute_code (自動化測試腳本)

# DB 測試
- npx supabase db query --linked
- Supabase REST API 查詢
```

## Phase 2: 紀錄 Document

### 標準紀錄格式

每個問題一張卡片：

```markdown
### 問題 #{n}: [簡短標題]
- **優先級**: P0/P1/P2
- **類型**: 前端顯示 / 後端邏輯 / DB限制 / 性能
- **發現位置**: 頁面 / 組件 / API
- **複現步驟**:
  1. 打開 ...
  2. 點擊 ...
  3. 顯示 ... (錯誤)
- **預期行為**: 應該 ...
- **實際行為**: 實際 ...
- **提示資訊**: Console 錯誤 / API 回應 / 截圖
- **根因分析**: [填寫]
- **修復方案**: [填寫]
```

### 建立 TODO

使用 `todo` 工具追蹤每個問題的狀態：
- `pending` → `in_progress` → `completed` / `cancelled`

## Phase 3: 規劃 Plan

### 根因分析模板

```
問題: [X] 發生了什麼？
↓
為什麼？ [Y] 哪個系統出錯？
↓
為什麼？ [Z] 系統的哪個部分有缺陷？
↓
修復方案: 修改 [Z]
```

### 優先級排序

| 優先級 | 標準 | 例子 |
|-------|------|-------|
| **P0** | 功能完全壞掉，用戶無法繼續 | 註冊 API 500 錯誤、登入失敗 |
| **P1** | 功能可用但體驗差，有 workaround | 按鈕無法點擊、滾動不正常 |
| **P2** | 文字、樣式、非關鍵問題 | 提示文字不清楚、顏色不一致 |

### 風險評估

修復前自問：
- 這個修改會影響哪些其他功能？
- 有無回歸風險？
- 是否需要 DB migration？
- 是否需要 feature flag？

---

## Phase 4: 修改 Fix

### 修改原則

1. **一次只修一個問題**，避免幾個修改互相干擾
2. **修改後立即測試**，不要等全部修完再測試
3. **DB 修復先於代碼修復**（若有依賴關係）
4. **保留原始行為**，只做最小必要的變更

### 修改檢查清單

```markdown
- [ ] 代碼修改完成
- [ ] DB 修復完成（如需）
- [ ] 註解更新
- [ ] npm run build 通過
- [ ] 本地測試通過（若可能）
- [ ] Deploy 到生產環境
```

---

## Phase 5: 再測試 Re-test

### 必測項目

1. **原問題是否解決**？（直接測試）
2. **修改區域附近功能是否正常**？（回歸）
3. **整體流程是否通暢**？（end-to-end）

### 測試結果記錄

```markdown
### 測試結果
- **測試時間**: 2026-04-18 14:00
- **測試環境**: https://clawvec.com (生產環境)
- **測試項目**: 導航列、AI 註冊、API 回應
- **結果**: 通過 / 失敗
- **備註**: [如有問題，記錄下來]
```

### 決策點

測試後：
- ✅ **全部通過** → 進入 Phase 6
- ❌ **還有問題** → 回到 Phase 3（規劃新修復方案）
- ⚠️ **發現新問題** → 回到 Phase 1（記錄新問題，排序優先級）

---

## Phase 6: 完成 Done

### 清理與文件化

1. **測試數據清理**
   - 刪除測試產生的假資料
   - 清理測試帳號、API key

2. **TODO 更新**
   - 所有項目標記 `completed`

3. **文件記錄**
   - 保存 SQL 修復腳本（若有）
   - 更新 CHANGELOG 或修復記錄

4. **Memory 更新**
   - 將發現的新陷阱、流程改進點記錄到 memory

5. **Skill 迭代**
   - 若流程有改進，更新本 skill

## 🧙 AI Agent 執行指南

### 自動化原則

- **主動繼續**：不要等用戶說"再測試"，修復後自動進入再測試
- **先完成再報告**：不要每修一個就打擾用戶，等所有 P0/P1 完成再統一報告
- **預設有問題**：修復後預設還有回歸問題，主動測試相關功能
- **清理是禮貌**：測試完畢一定要清理測試數據

### 檢查點清單

每次部署後必須檢查：
- [ ] 頁面能正常加載
- [ ] 瀏覽器 Console 無紅色錯誤
- [ ] 核心功能流程通過
- [ ] 行動裝置顯示正常
- [ ] 登入/登出狀態正常
